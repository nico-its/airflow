# Guide complet d'utilisation d'Apache Airflow
## Formation Lab — Utilisation d’Airflow

---

# 1. Introduction

Apache Airflow est une plateforme open source permettant d’orchestrer des workflows.

Un workflow correspond à une suite de traitements automatisés exécutés dans un ordre précis.

Airflow est principalement utilisé pour :

- orchestrer des pipelines de données
- automatiser des traitements techniques
- exécuter des scripts dans un ordre défini
- gérer des dépendances entre tâches
- surveiller l’exécution des jobs

Airflow n’exécute **pas directement la logique métier**.

Il orchestre l’exécution de :

- scripts shell
- programmes Python
- requêtes SQL
- conteneurs
- appels API
- traitements ETL

---

# 2. Principe de fonctionnement

Dans Airflow un workflow est appelé :

## DAG — Directed Acyclic Graph

| Terme | Signification |
|------|------|
| Directed | les tâches ont un ordre d'exécution |
| Acyclic | aucune boucle n’est possible |
| Graph | les tâches forment un graphe |

Un DAG est composé de :

```
DAG
 ├── tâches
 ├── dépendances
 ├── planification
 └── configuration
```

---

# 3. Architecture d’Airflow

Airflow repose sur plusieurs composants.

```
                +-------------+
                | Webserver   |
                +-------------+
                       |
                       |
                +-------------+
                | Scheduler   |
                +-------------+
                       |
                       |
                +-------------+
                | Executor    |
                +-------------+
                       |
                       |
                +-------------+
                | Workers     |
                +-------------+

                +------------------+
                | Metadata DB      |
                +------------------+
```

---

## 3.1 Scheduler

Le scheduler est le moteur d’orchestration.

Fonctions :

1. Analyse les fichiers DAG
2. Crée les DAG Runs
3. Détermine quelles tâches sont exécutables
4. Envoie les tâches à l'executor

Le scheduler fonctionne en continu.

---

## 3.2 Webserver

Le Webserver fournit l’interface graphique.

Fonctionnalités :

- visualiser les DAGs
- analyser les graphes
- consulter les logs
- déclencher des DAGs
- relancer des tâches
- analyser les erreurs

---

## 3.3 Metadata Database

La base de données Airflow stocke :

- DAG Runs
- Task Instances
- Variables
- Connections
- Pools
- Datasets
- Logs

Bases généralement utilisées :

- PostgreSQL
- MySQL

---

## 3.4 Executor

L’executor exécute réellement les tâches.

| Executor | Description |
|------|------|
LocalExecutor | exécution locale |
CeleryExecutor | workers distribués |
KubernetesExecutor | pods Kubernetes |

# 4. Structure d’un DAG

Un DAG est un fichier Python.

Exemple minimal :

```python
from airflow import DAG
from datetime import datetime

with DAG(
    dag_id="example_dag",
    start_date=datetime(2024,1,1),
    schedule="@daily",
    catchup=False
) as dag:

    pass
```

---

## Paramètres importants

### dag_id

Identifiant unique du DAG.

### start_date

Date de début d’exécution.

### schedule

Fréquence d’exécution.

Exemples :

```
@daily
@hourly
@weekly
```

Ou cron :

```
0 2 * * *
```

### catchup

Si activé : Airflow exécute les DAG runs manqués.

---

# 5. DAG Run

Un DAG Run correspond à une exécution d’un DAG.

Exemple :

```
DAG : pipeline_daily

DAG Runs :

2024-01-01
2024-01-02
2024-01-03
```

Chaque DAG Run possède :

- des task instances
- des logs
- un statut


# 8. Cycle de vie d’une tâche

Une tâche peut avoir plusieurs états.

```
none

failed

```

Diagramme simplifié :

```
scheduled
   |

   |

```

# 9. Trigger Rules

Les trigger rules définissent quand une tâche peut s’exécuter.

| Trigger Rule | Description |
|------|------|
all_success | toutes les tâches précédentes réussissent |
all_failed | toutes les tâches échouent |
one_success | au moins une réussit |
one_failed | au moins une échoue |
none_failed | aucune échouée |
none_skipped | aucune ignorée |
always | toujours exécutée |

# 10. Pools

Les pools permettent de limiter la concurrence.

Utilisation :

- limiter la charge d'une base de données
- protéger une API
- limiter l'utilisation CPU

---

## Exemple

```
pool : database_pool
slots : 3
```

Cela signifie :

```
maximum 3 tâches simultanées
```

---

## Fonctionnement interne

Le scheduler :

1. vérifie le pool
2. regarde les slots disponibles
3. si aucun slot disponible

```
la tâche reste en état queued
```

---

## Exemple dans un DAG

```python
task = BashOperator(
    task_id="query_database",
    bash_command="script.sh",
    pool="database_pool"
)
```

---

# 11. Priorité des tâches

Chaque tâche possède un paramètre :

```
priority_weight
```

Plus la valeur est élevée, plus la tâche est prioritaire.

Exemple :

```
task1 priority_weight = 1
task2 priority_weight = 10
```

task2 sera exécutée en premier.

## 1. Principe des connexions Airflow

Une connexion Airflow permet de stocker les informations nécessaires pour accéder à une ressource externe.

Une connexion peut contenir :

- host
- port
- login
- password
- schema
- chemin filesystem
- paramètres supplémentaires

Les connexions sont configurées dans l’interface :

```
Admin → Connections
```

Chaque connexion possède un identifiant unique appelé :

```
Connection ID
```

Cet identifiant est ensuite utilisé dans les DAGs.

---

## 2. Paramètres de connexion dans les DAGs

Selon le type d’opérateur ou de sensor utilisé, le paramètre change.

| Type d’intégration | Paramètre utilisé |
|---|---|
Filesystem | fs_conn_id |
PostgreSQL | postgres_conn_id |
HTTP | http_conn_id |
AWS | aws_conn_id |
SQL générique | conn_id |

Il est donc nécessaire de vérifier la signature de l’opérateur utilisé.

---

## 3. Connexion Filesystem pour FileSensor

### Création de la connexion

Dans Airflow :

```
Admin → Connections → Add Connection
```

Exemple de configuration :

```
Connection Id : lab_fs
Connection Type : Filesystem
Host / Path : /tmp/airflow_lab
```

Cette valeur représente la racine du filesystem utilisé par le sensor.

---

### Exemple d’utilisation dans un DAG

```python
from airflow import DAG
from airflow.providers.standard.sensors.filesystem import FileSensor
from airflow.providers.standard.operators.bash import BashOperator
from datetime import datetime

with DAG(
    dag_id="example_filesensor_dag",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False
) as dag:

    wait_for_file = FileSensor(
        task_id="wait_for_file",
        fs_conn_id="lab_fs",
        filepath="input/customers.csv",
        poke_interval=30,
        timeout=600,
        mode="reschedule"
    )

    process_file = BashOperator(
        task_id="process_file",
        bash_command="echo 'Fichier détecté'"
    )

    wait_for_file >> process_file
```

---

### Résolution du chemin

Si la connexion pointe vers :

```
/tmp/airflow_lab
```

et que le DAG utilise :

```
filepath="input/customers.csv"
```

Airflow vérifiera :

```
/tmp/airflow_lab/input/customers.csv
```

## 5. Utilisation PostgreSQL dans un DAG

### Exemple avec SQLExecuteQueryOperator

```python
from airflow import DAG
from airflow.providers.common.sql.operators.sql import SQLExecuteQueryOperator
from datetime import datetime

with DAG(
    dag_id="postgres_query_example",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False
) as dag:

    query_database = SQLExecuteQueryOperator(
        task_id="query_database",
        conn_id="pg_lab",
        sql="SELECT current_database();"
    )
```

---

### Exemple avec PostgresHook

```python
from airflow import DAG
from airflow.providers.standard.operators.python import PythonOperator
from airflow.providers.postgres.hooks.postgres import PostgresHook
from datetime import datetime

def read_database():
    hook = PostgresHook(postgres_conn_id="pg_lab")
    result = hook.get_records("SELECT id, name FROM customers LIMIT 5;")
    for row in result:
        print(row)

with DAG(
    dag_id="postgres_hook_example",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False
) as dag:

    read_data = PythonOperator(
        task_id="read_data",
        python_callable=read_database
    )
```

---

### Exemple avec SqlSensor

```python
from airflow import DAG
from airflow.providers.common.sql.sensors.sql import SqlSensor
from airflow.providers.standard.operators.bash import BashOperator
from datetime import datetime

with DAG(
    dag_id="sql_sensor_example",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False
) as dag:

    wait_for_data = SqlSensor(
        task_id="wait_for_data",
        conn_id="pg_lab",
        sql="SELECT 1 FROM customers WHERE status='READY' LIMIT 1;",
        poke_interval=30,
        timeout=600,
        mode="reschedule"
    )

    continue_process = BashOperator(
        task_id="continue_process",
        bash_command="echo 'Données disponibles'"
    )

    wait_for_data >> continue_process
```

## 7. Principe des Assets

Les Assets permettent de représenter une donnée produite par un DAG.

Un Asset peut représenter :

- un fichier
- une table
- un dataset
- une ressource métier

Exemples :

```
customers_dataset
orders_table
file:///tmp/airflow_lab/input/customers.csv
```

Les Assets permettent de déclencher un DAG lorsqu’une donnée est mise à jour.

---

## 8. Déclaration d’un Asset

```python
from airflow.sdk import Asset

customers_asset = Asset("file:///tmp/airflow_lab/input/customers.csv")
```

---

## 9. DAG producteur utilisant un Asset

```python
from airflow import DAG
from airflow.providers.standard.operators.bash import BashOperator
from airflow.sdk import Asset
from datetime import datetime

customers_asset = Asset("file:///tmp/airflow_lab/input/customers.csv")

with DAG(
    dag_id="produce_customers_file",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False
) as dag:

    generate_file = BashOperator(
        task_id="generate_file",
        bash_command="""
        mkdir -p /tmp/airflow_lab/input
        cat > /tmp/airflow_lab/input/customers.csv <<EOF
id,name,city
1,Alice,Paris
2,Bob,Lyon
EOF
        """,
        outlets=[customers_asset]
    )
```

Dans cet exemple :

```
outlets=[customers_asset]
```

signifie que la tâche publie une mise à jour de cet Asset.

---

## 10. DAG consommateur basé sur un Asset

```python
from airflow import DAG
from airflow.providers.standard.operators.bash import BashOperator
from airflow.sdk import Asset
from datetime import datetime

customers_asset = Asset("file:///tmp/airflow_lab/input/customers.csv")

with DAG(
    dag_id="consume_customers_file",
    start_date=datetime(2024, 1, 1),
    schedule=[customers_asset],
    catchup=False
) as dag:

    process_file = BashOperator(
        task_id="process_file",
        bash_command="""
        echo "Lecture du fichier produit"
        cat /tmp/airflow_lab/input/customers.csv
        """
    )
```

Dans cet exemple :

```
schedule=[customers_asset]
```

signifie que le DAG démarre lorsque cet Asset est mis à jour.

## 12. Différence entre FileSensor et Asset

| Mécanisme | Usage |
|---|---|
FileSensor | attendre physiquement un fichier |
Asset | modéliser une donnée produite |

FileSensor :

```
attend que le fichier existe
```

Asset :

```
déclare qu’une donnée est mise à jour
```
