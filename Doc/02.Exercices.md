## Exercice 1 — Structure minimale d'un DAG

### Objectif
Créer le squelette d'un DAG Airflow sans aucune tâche réelle, juste pour comprendre les paramètres obligatoires.

### Étapes
1. Créer un fichier `dag_ex1.py` dans le dossier `dags/`.
2. Importer `DAG` depuis `airflow` et `datetime` depuis `datetime`.
3. Instancier un DAG avec les paramètres suivants :
   - `dag_id` : `"exercice_1"`
   - `start_date` : 1er janvier 2024
   - `schedule` : `"@daily"`
   - `catchup` : `False`
4. Vérifier que le DAG apparaît dans l'interface Airflow sans erreur.

### Rappels
- `catchup=False` évite de rejouer toutes les exécutions passées au démarrage.
- Le `dag_id` doit être **unique** dans tout Airflow.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from datetime import datetime

with DAG(
    dag_id="exercice_1",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:
    pass  # Aucune tâche pour l'instant
```

</details>

## Exercice 2 — Première tâche avec `EmptyOperator`

### Objectif
Ajouter des tâches "vides" pour représenter un flux de travail sans logique métier, utile pour tester les enchaînements.

### Étapes
1. Repartir du DAG de l'exercice 1.
2. Importer `EmptyOperator` depuis `airflow.operators.empty`.
3. Créer 3 tâches : `start`, `processing`, `end`.
4. Enchaîner les tâches dans cet ordre : `start → processing → end`.
5. Vérifier le graphe dans l'UI Airflow (onglet **Graph**).

### Rappels
- L'opérateur `>>` définit une dépendance : `t1 >> t2` signifie "t2 s'exécute après t1".
- `EmptyOperator` ne fait rien mais permet de structurer visuellement un DAG.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.operators.empty import EmptyOperator
from datetime import datetime

with DAG(
    dag_id="exercice_2",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:

    start = EmptyOperator(task_id="start")
    processing = EmptyOperator(task_id="processing")
    end = EmptyOperator(task_id="end")

    start >> processing >> end
```

</details>

## Exercice 3 — Exécuter des commandes shell avec `BashOperator`

### Objectif
Utiliser `BashOperator` pour exécuter des commandes shell depuis un DAG.

### Étapes
1. Créer un nouveau DAG `exercice_3` avec un `schedule="@once"`.
2. Importer `BashOperator` depuis `airflow.operators.bash`.
3. Créer 2 tâches :
   - `print_date` : exécute la commande `date`
   - `print_message` : exécute `echo "Pipeline terminé avec succès"`
4. Enchaîner : `print_date → print_message`.
5. Déclencher manuellement le DAG et consulter les **logs** de chaque tâche.

### Rappels
- Le paramètre `bash_command` attend une chaîne de caractères shell.
- Les logs s'affichent en cliquant sur une tâche → **Log** dans l'UI.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime

with DAG(
    dag_id="exercice_3",
    start_date=datetime(2024, 1, 1),
    schedule="@once",
    catchup=False,
) as dag:

    print_date = BashOperator(
        task_id="print_date",
        bash_command="date",
    )

    print_message = BashOperator(
        task_id="print_message",
        bash_command="echo 'Pipeline terminé avec succès'",
    )

    print_date >> print_message
```

</details>

## Exercice 4 — Logique Python avec `PythonOperator`

### Objectif
Exécuter des fonctions Python directement dans un DAG via `PythonOperator`.

### Étapes
1. Créer un DAG `exercice_4` planifié `"@daily"`.
2. Importer `PythonOperator` depuis `airflow.providers.standard.operators.python`.
3. Définir deux fonctions Python **en dehors** du bloc `with DAG` :
   - `extraire_donnees()` : affiche `"Extraction en cours..."`
   - `transformer_donnees()` : affiche `"Transformation en cours..."`
4. Créer une tâche par fonction via `PythonOperator` (paramètre `python_callable`).
5. Enchaîner : `extraire → transformer`.

### Rappels
- `python_callable` reçoit la **référence** de la fonction (sans parenthèses).
- Les fonctions doivent être définies **avant** le bloc `with DAG`.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.providers.standard.operators.python import PythonOperator
from datetime import datetime

def extraire_donnees():
    print("Extraction en cours...")

def transformer_donnees():
    print("Transformation en cours...")

with DAG(
    dag_id="exercice_4",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:

    extraire = PythonOperator(
        task_id="extraire_donnees",
        python_callable=extraire_donnees,
    )

    transformer = PythonOperator(
        task_id="transformer_donnees",
        python_callable=transformer_donnees,
    )

    extraire >> transformer
```

</details>

## Exercice 5 — Dépendances parallèles

### Objectif
Créer un DAG avec des tâches s'exécutant **en parallèle**, puis convergeant vers une tâche finale.

### Étapes
1. Créer un DAG `exercice_5`.
2. Créer les tâches suivantes avec `EmptyOperator` :
   - `start`
   - `traitement_a` et `traitement_b` (parallèles)
   - `fin`
3. Définir les dépendances :
   - `start` déclenche **simultanément** `traitement_a` et `traitement_b`
   - `fin` attend que **les deux** soient terminées
4. Vérifier le graphe : il doit ressembler à un losange.

### Rappels
- `t1 >> [t2, t3]` lance t2 et t3 en parallèle après t1.
- `[t2, t3] >> t4` fait attendre t4 que t2 et t3 soient terminées.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.operators.empty import EmptyOperator
from datetime import datetime

with DAG(
    dag_id="exercice_5",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:

    start = EmptyOperator(task_id="start")
    traitement_a = EmptyOperator(task_id="traitement_a")
    traitement_b = EmptyOperator(task_id="traitement_b")
    fin = EmptyOperator(task_id="fin")

    start >> [traitement_a, traitement_b] >> fin
```

</details>

## Exercice 6 — Utiliser des `Variables` Airflow

### Objectif
Paramétrer un DAG en lisant des valeurs stockées dans les **Variables Airflow** (Admin → Variables).

### Étapes
1. Dans l'UI Airflow, aller dans **Admin → Variables** et créer :
   - `environment` = `"production"`
   - `data_path` = `"/data/input"`
2. Créer un DAG `exercice_6`.
3. Importer `Variable` depuis `airflow.models`.
4. Dans une fonction Python, lire les deux variables et les afficher dans les logs.
5. Créer un `PythonOperator` qui appelle cette fonction.

### Rappels
- `Variable.get("ma_variable")` retourne la valeur sous forme de chaîne.
- Ne jamais stocker de credentials directement dans le code du DAG.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.models import Variable
from airflow.providers.standard.operators.python import PythonOperator
from datetime import datetime

def afficher_config():
    env = Variable.get("environment")
    path = Variable.get("data_path")
    print(f"Environnement : {env}")
    print(f"Chemin des données : {path}")

with DAG(
    dag_id="exercice_6",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:

    afficher = PythonOperator(
        task_id="afficher_config",
        python_callable=afficher_config,
    )
```

</details>

## Exercice 7 — Gérer les échecs avec les `Trigger Rules`

### Objectif
Comprendre comment contrôler l'exécution d'une tâche même si une tâche amont a échoué, grâce aux **Trigger Rules**.

### 📋 Étapes
1. Créer un DAG `exercice_7`.
2. Créer 3 tâches :
   - `tache_risquee` : un `BashOperator` qui exécute `exit 1` (force l'échec)
   - `tache_normale` : un `BashOperator` affichant `"Exécution normale"`
   - `nettoyage` : un `BashOperator` affichant `"Nettoyage toujours exécuté"`
3. Configurer `nettoyage` avec `trigger_rule="all_done"` pour qu'elle s'exécute **quoi qu'il arrive**.
4. Enchaîner : `tache_risquee` et `tache_normale` → `nettoyage`.
5. Déclencher le DAG et observer les états des tâches dans l'UI.

### Rappels des Trigger Rules principales

| Règle | Comportement |
|---|---|
| `all_success` *(défaut)* | S'exécute si toutes les tâches amont ont réussi |
| `all_failed` | S'exécute si toutes les tâches amont ont échoué |
| `one_success` | S'exécute dès qu'une tâche amont réussit |
| `none_failed` | S'exécute si aucune tâche amont n'a échoué |
| `all_done` | S'exécute quand toutes les tâches amont sont terminées (succès ou échec) |

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.operators.bash import BashOperator
from airflow.utils.trigger_rule import TriggerRule
from datetime import datetime

with DAG(
    dag_id="exercice_7",
    start_date=datetime(2024, 1, 1),
    schedule="@once",
    catchup=False,
) as dag:

    tache_risquee = BashOperator(
        task_id="tache_risquee",
        bash_command="exit 1",
    )

    tache_normale = BashOperator(
        task_id="tache_normale",
        bash_command="echo 'Exécution normale'",
    )

    nettoyage = BashOperator(
        task_id="nettoyage",
        bash_command="echo 'Nettoyage toujours exécuté'",
        trigger_rule=TriggerRule.ALL_DONE,
    )

    [tache_risquee, tache_normale] >> nettoyage
```

</details>

## Exercice 8 — Attendre un fichier avec `FileSensor`

### Objectif
Utiliser un **Sensor** pour bloquer l'exécution d'un pipeline jusqu'à ce qu'un fichier soit présent sur le système de fichiers.

### 📋 Étapes
1. Dans l'UI Airflow, créer une **Connexion** (Admin → Connections) :
   - **Connection Id** : `lab_fs`
   - **Connection Type** : `File (path)`
   - **Host** : `/tmp`
2. Créer un DAG `exercice_8` avec `schedule=None`.
3. Importer `FileSensor` depuis `airflow.providers.standard.sensors.filesystem`.
4. Créer une tâche `wait_for_file` qui attend le fichier `input/customers.csv` avec :
   - `poke_interval=30` (vérifie toutes les 30s)
   - `timeout=600` (abandon après 10 min)
   - `mode="reschedule"` (libère le worker entre deux vérifications)
5. Créer une tâche `process_file` (BashOperator) qui affiche `"Fichier détecté !"`.
6. Enchaîner : `wait_for_file → process_file`.

### Rappels
- `mode="reschedule"` est préférable à `"poke"` pour ne pas monopoliser un worker.
- Le chemin final résolu est : `host` (connexion) + `filepath` (sensor).

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.providers.standard.sensors.filesystem import FileSensor
from airflow.operators.bash import BashOperator
from datetime import datetime

with DAG(
    dag_id="exercice_8",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False,
) as dag:

    wait_for_file = FileSensor(
        task_id="wait_for_file",
        fs_conn_id="lab_fs",
        filepath="input/customers.csv",
        poke_interval=30,
        timeout=600,
        mode="reschedule",
    )

    process_file = BashOperator(
        task_id="process_file",
        bash_command="echo 'Fichier détecté !'",
    )

    wait_for_file >> process_file
```

</details>

## Exercice 9 — Limiter la concurrence avec les `Pools`

### Objectif
Utiliser un **Pool** pour limiter le nombre de tâches s'exécutant simultanément sur une ressource partagée (ex : base de données).

### 📋 Étapes
1. Dans l'UI Airflow, créer un Pool (Admin → Pools) :
   - **Pool** : `database_pool`
   - **Slots** : `3`
2. Créer un DAG `exercice_9` avec `schedule="@daily"`.
3. Créer 5 tâches `PythonOperator` simulant des accès base de données (chacune affiche son nom).
4. Assigner **toutes** les tâches au pool `database_pool` via le paramètre `pool`.
5. Lancer le DAG et observer dans l'UI que **3 tâches maximum** tournent simultanément.

### Rappels
- Le paramètre `pool` est disponible sur tous les opérateurs.
- Si tous les slots sont pris, les tâches restent en état `queued` jusqu'à libération.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.providers.standard.operators.python import PythonOperator
from datetime import datetime

def acces_bdd(nom_tache):
    import time
    print(f"Accès base de données : {nom_tache}")
    time.sleep(5)

with DAG(
    dag_id="exercice_9",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:

    taches = []
    for i in range(1, 6):
        t = PythonOperator(
            task_id=f"acces_bdd_{i}",
            python_callable=acces_bdd,
            op_kwargs={"nom_tache": f"tache_{i}"},
            pool="database_pool",
        )
        taches.append(t)
```

</details>

## Exercice 10 — DAGs communicants via les `Assets`

### Objectif
Créer deux DAGs **découplés** : un **producteur** qui génère un fichier et un **consommateur** qui se déclenche automatiquement dès que le fichier est produit, grâce au mécanisme d'**Assets**.

### 📋 Étapes

**DAG Producteur :**
1. Déclarer un Asset : `Asset("file:///tmp/airflow_lab/input/customers.csv")`.
2. Créer un DAG `produce_customers_file` avec `schedule=None`.
3. Ajouter un `BashOperator` qui crée le fichier :
   ```bash
   mkdir -p /tmp/airflow_lab/input && echo 'id,name' > /tmp/airflow_lab/input/customers.csv
   ```
4. Déclarer que cette tâche **produit** l'Asset via `outlets=[customers_asset]`.

**DAG Consommateur :**

5. Créer un DAG `consume_customers_file` avec `schedule=[customers_asset]`.
6. Ajouter un `BashOperator` qui lit le fichier : `cat /tmp/airflow_lab/input/customers.csv`.

**Test :**

7. Déclencher manuellement le DAG producteur et observer le DAG consommateur se lancer **automatiquement**.

### Rappels
- `outlets` sur un opérateur signale qu'une tâche **met à jour** un Asset.
- `schedule=[asset]` configure le DAG pour se déclencher **à la mise à jour** de cet Asset.
- Ce mécanisme remplace avantageusement `TriggerDagRunOperator` pour découpler les pipelines.

<details>
<summary>Solution</summary>

```python
from airflow import DAG
from airflow.sdk import Asset
from airflow.operators.bash import BashOperator
from datetime import datetime

# Déclaration de l'Asset partagé entre les deux DAGs
customers_asset = Asset("file:///tmp/airflow_lab/input/customers.csv")

# --- DAG Producteur ---
with DAG(
    dag_id="produce_customers_file",
    start_date=datetime(2024, 1, 1),
    schedule=None,
    catchup=False,
) as dag_producer:

    generate_file = BashOperator(
        task_id="generate_file",
        bash_command="mkdir -p /tmp/airflow_lab/input && echo 'id,name\n1,Alice\n2,Bob' > /tmp/airflow_lab/input/customers.csv",
        outlets=[customers_asset],
    )

# --- DAG Consommateur ---
with DAG(
    dag_id="consume_customers_file",
    start_date=datetime(2024, 1, 1),
    schedule=[customers_asset],
    catchup=False,
) as dag_consumer:

    read_file = BashOperator(
        task_id="read_file",
        bash_command="cat /tmp/airflow_lab/input/customers.csv",
    )
```

</details>
